/# Commit Rules

## 기본 원칙

- 커밋 전 반드시 `git status`와 `git diff`를 확인한다.
- 현재 작업 목적과 무관한 파일 변경은 커밋에 포함하지 않는다.
- 한 커밋은 하나의 의도(하나의 논리 단위)만 담는다.

## 네이밍 규칙

- 브랜치: `타입/이슈번호-작업-내용`
  - 예: `feat/12-login-api`, `fix/2-notion-db-alias-compat`
- 커밋 메시지: `타입: 이모티콘 작업 내용`
  - 예: `feat: ✨ 로그인 API 구현`
  - 예: `fix: 🔨 Notion DB 스키마 자동 매핑 처리`

### 타입/이모티콘 매핑

- `feat`: ✨
- `fix`: 🔨
- `chore`: 🧹
- `docs`: 📝
- `refactor`: ♻️

## 체크리스트

- [ ] 변경 파일이 이번 작업 범위에 정확히 맞는다.
- [ ] 워크플로우/템플릿 변경 시 YAML 문법 검증을 수행했다.
- [ ] 스크립트 변경 시 문법 체크를 수행했다. (`node --check ...`)
- [ ] 필요한 경우 테스트를 실행했다. (`./gradlew test` 또는 관련 테스트)

## 금지 사항

- unrelated 변경사항을 함께 커밋하지 않는다.
- 임시 디버깅 코드/불필요 로그를 남기지 않는다.
- 실패한 상태를 숨기기 위해 테스트를 임의로 비활성화하지 않는다.
